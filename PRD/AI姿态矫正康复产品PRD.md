> **用途**：供编码 Agent、测试 Agent、评审 Agent 在不全文扫描 1700+ 行时仍能正确实现 MVP。

### 0.5.5 模块边界与仓库约定（TBD 路径）

## 2. 产品概述

### 2.1 一句话定义

一套围绕「办公坐姿」的多点 IMU 外设 + 端侧 Qwen 模型（MNN 推理）+ 云端 Qwen-VL 的混合 AI 姿态康复系统。MVP 阶段只做一件事：在用户久坐时，**帮助他保持「不驼背」的坐姿**——通过持续监测胸椎和头部的相对角度、端侧 AI 实时推理分类、温柔地提醒、并用 5-10 分钟的针对性训练帮用户建立习惯。

### 2.2 核心价值

现有轻量姿态产品多为单点传感器提醒，只能告诉用户「你弯了」。本产品通过多点 IMU 追踪脊柱关键节段，端侧 Qwen3.5-2B（MNN 推理）做实时姿态分类与本地反馈，云端 Qwen3.6-Plus 做视觉评估与报告生成，形成「检测 - 提醒 - 训练 - 复盘」闭环。核心交互逻辑 100% 本地运行，视觉评估走云端，云+端混合架构。

### 2.3 产品边界

本产品定位为健康管理与运动康复辅助工具，不作为医疗诊断设备，不输出疾病诊断、处方治疗或替代医生判断的结论。涉及疼痛、术后、严重脊柱侧弯等场景时，产品只提供就医提示与低风险训练建议。

#### US-005: 跟练康复训练

#### US-006: 查看日报

**描述：** 作为用户，我希望每天查看坐姿日报，以便了解今日表现和改进方向。

**验收标准：**

| 久坐        | 连续佩戴 > 45 分钟         | P1      | 提示「该起身了」，不进入异常事件聚合 |


### 5.5.3 数据主线

- 仪表盘主指标：「不驼背分数」（0-100，定义为「过去 60 分钟内不驼背的时长占比」加权调整）。
- 周报主图：「不驼背时长 / 日」折线。
- 训练计划问题倾向：仅从 {头前倾, 圆肩, 胸椎后凸} 三类生成。
- 疼痛反馈：若用户对某动作标记疼痛，该动作永久停推。

### 5.5.4 硬件主线

- 3 节点硬件保留：初赛用 Node-T 单节点驱动主指标，Node-C / Node-L BLE 链路打通但 App 不解析（复赛补全判定，见 §5.8.1 #2、§0.5.2 裁决表）。
- 不为此决策调整 BOM：3 节点背带是平台能力，复赛扩展时不用重做硬件。

### 5.5.5 不在 MVP 范围

- 站立 / 行走 / 驾驶 / 健身动作监测。
- 腰椎塌陷与左右倾斜的 P0 级判定与 UI 提示。
- 多人 / 多设备协同。
- 训练动作识别与跟练质量评分。
- 端侧 **Qwen3-VL-2B 及以上**视觉大模型（初赛用 Qwen3.5-2B 文本模型，见 F9）。
- 任何医疗级表述与承诺（见[技术规格文档 §3.2](技术规格文档.md#32-安全限制安全链)）。

如果评审中有人提出扩展到「全姿态」或「健身动作监测」，按 §5.4 显式 Out-of-Scope 处理，并在 0 节变更记录中登记。

## 5.6 竞品参考：NeckGo 启示与我们的取舍（v1.4）

NeckGo 是 iOS 上一款以 AirPods 头部追踪为核心、靠游戏化留住用户的"颈椎微度假"产品。它的设计和技术选型对我们有借鉴价值，但路径不同（它没有外设硬件，我们有 3 节点 IMU 背带）。下表是逐项的取舍：

### 5.6.1 我们要学的（A 档）


| NeckGo 做法                                | 我们的对应做法                                      | 理由                               |
| ---------------------------------------- | -------------------------------------------- | -------------------------------- |
| 进场流程 3 步：检测 AirPods → 一键校准 → 启动监测        | §6.1 首次流程压成 3 步：连接 → 校准 → 进入首页               | 演示需要"快速看见脊柱"，3 步到位比 10 步流程更抓人    |
| "轻拍肩膀"的震感，< 200ms 单次，靠 iOS Taptic Engine | §11.3 设计原则加"震感克制"：单次 < 200ms、间隔 ≥ 10 分钟、强度三档 | 提醒是"温和陪伴"不是"教官命令"，与 v1.3 R8 风险呼应 |
| 状态色克制，全程不用大面积红色                          | §11.3 设计原则加"状态色克制"：红色仅用于持续异常提醒               | 与 2 人 doc「红色不滥用」一脉相承，写进 P0 硬规则   |
| 零位校准思维，校准 = "现在坐姿 = 基准"                  | §F2 校准主路径压成 30 秒坐姿基线，视觉评估改为可选                | 减少首次摩擦，同时保留视觉修正量作为可选 P1          |


### 5.6.2 视情况采用（B 档）

> **初赛状态（v1.5）**：本节持续通知 / 桌面小组件初赛不做；校准 = 一次点击，初赛走 5 秒快速校准版本（不做 30 秒）。


| NeckGo 做法                 | 我们的对应做法                                                      | 理由                                    |
| ------------------------- | ------------------------------------------------------------ | ------------------------------------- |
| Live Activities 灵动岛长驻状态显示 | Android 等价物：持续通知（Ongoing Notification），列为复赛 P0（§11.2、§5.8.2） | 用户不在主页时也能瞥到"现在驼了没"，演示时可指给评委看；**初赛不做** |
| 桌面小组件（StandBy 模式）         | Android 等价物：App Widget，列为 P1（§11.2）                          | 加分项，4 周内若排得开再做                        |
| 校准 = 一次点击                 | §F2 校准默认 30 秒，提供「快速校准 5 秒」选项                                 | 30 秒在演示里偏长，给评委"一键校准"的快路径              |


### 5.6.3 明确不学（C 档）


| NeckGo 做法            | 不学的理由                               |
| -------------------- | ----------------------------------- |
| 果实收集、地图解锁、角色养成等完整游戏化 | 我们是健康工具不是游戏，4 周排期吃不消，评委也不会因为"好玩"打高分 |
| 空间音频 / 白噪音 / 3D 音效   | 评审现场大多没戴耳机，演示场景不友好                  |
| Apple Watch 联动       | 我们做 Android，等价的是 Wear OS，P2 候补      |
| Vision / ARKit 摄像头视觉 | 我们有 3 节点 IMU，硬件是差异化的根本，相机方案会模糊叙事    |


### 5.6.4 候补 / P2（D 档）

- Wear OS 端联动：增加 Android 生态完整度
- App 图标角标显示今日分数
- 桌面小组件显示"今日不驼背时长 + 小植物状态"

## 5.7 轻量游戏化 3 件套（v1.4）

> **初赛状态**：植物 / Streak / 徽章初赛做（§5.7.1—5.7.3），仪表盘集成。

| 徽章    | 解锁条件                 |

- 随机果实掉落、地图解锁、角色养成、强制每日任务

## 5.10 AI vs 规则的边界（为什么不用纯规则）

> **核心问题**：姿态分类本质是阈值判断，规则引擎足够，为什么要上端侧大模型？是不是为了 AI 而 AI？

### 哪些任务用规则（不需要 AI）

| 任务 | 实现方式 | 理由 |
|------|---------|------|
| 姿态分类（驼背/头前倾/正常） | F3 状态机 + 阈值 | 角度超阈值就判定，规则确定性强、0 延迟、0 内存 |
| 提醒触发与冷却 | F3 状态机 | 计时器 + 状态迁移，规则引擎最可靠 |
| 不驼背分数计算 | 公式：`score = 100 - 异常时长占比 × 权重` | 纯数学，不需要 AI |

**结论**：分类和提醒用规则，不依赖 AI。这是产品的可靠性底线。

### 哪些任务必须用 AI（规则做不到）

| 任务 | 为什么规则不行 | AI 方案 |
|------|--------------|---------|
| 本地反馈文案生成 | 规则只能输出"你驼背了"，无法根据上下文生成个性化建议（如"头部前移约15°，建议做颈部回缩训练"） | 端侧 Qwen3.5-2B 输入角度序列 + 用户状态 → 输出自然语言建议 |
| 视觉体态评估 | 规则无法理解照片中的体态特征 | 云端 Qwen3.6-Plus 输入侧身照 → 输出结构化观察 |
| 训练计划个性化 | 规则只能查表，无法根据用户历史+禁忌生成适配方案 | 云端/端侧 Qwen 生成（初赛走预置，复赛接 AI） |

### 赛事硬要求

赛事评分维度（§5.9）明确要求：
- **创新性（30%）**：端侧 Qwen3.5-2B（MNN）实时推理 + 云端 Qwen3.6-Plus 视觉评估
- **技术水平（20%）**：MNN 框架 + Arm SME2 适配 + Qwen INT4 量化

不用 Qwen = 创新性维度直接丢分。

### 防止"为了 AI 而 AI"的 3 条纪律

1. **分类不走 AI**：F3 状态机用规则，不用模型。模型只做规则做不好的事（文案生成）。
2. **AI 必须有回退**：F9 推理超时/崩溃 → 自动降级到规则引擎，主流程不中断。
3. **AI 输出必须过安全链**：模型输出结构化字段 → 本地查表翻译 → 禁词检查，不允许模型直接输出给用户。

## 6. 用户流程

### 6.1 首次使用流程（v1.4 简化为 3 步，v1.5 初赛版进一步精简）

> **硬约束**：从打开 App 到进入主仪表盘 **≤ 30 秒**。所有"细致但不影响首屏可见价值"的步骤都收进二级流程，由用户主动进入。
> **初赛状态（v1.5）**：30 秒完整校准改为 5 秒快速校准（按下"开始"后倒计时 5 秒即完成）；视觉评估照片环节初赛整体不出现。

主流程（必走，3 步）：

1. **连接外设**：App 启动后自动检测已配对的 PostureBand，未配对则引导扫描。连接成功即视为完成。
2. **校准基线**（v1.5 初赛 5 秒版）：提示"保持你自然的坐姿"，5 秒倒计时采集完成，App 自动记录基线。视觉评估照片初赛不出现，留到复赛。
3. **进入主仪表盘**：展示「不驼背分数 + 脊柱可视化 + 1 节点连接状态 + 小植物」，结束首次流程。

二级流程（首次可不走，之后随时可进）：

- A. 选目的（办公坐姿 / 体态改善 / 康复辅助）：首次进入仪表盘后弹 1 次，3 选 1 + 跳过。
- B. 详细传感器自检：设备页 → "自检"按钮。
- C. 视觉体态评估：仪表盘右上角"重新校准" → "用照片校准"。
- D. 佩戴引导视频：设备页 → "如何佩戴"按钮。
- E. 安全说明全文：设置 → "关于" → "非医疗声明"。
- F. 训练计划生成：仪表盘 → "今日训练"自动基于已选目的 + 校准数据生成，无需首屏手点。

演示场景下，评委看到的链路是：打开 App（0:00）→ 自动检测已配对外设（0:05）→ **5 秒**快速校准（0:10 完成）→ 看到脊柱 + 不驼背分数（0:15）。中间所有弹窗都被收起来了，总时长 **≤ 30 秒**（§6.1 硬约束）。

### 6.2 日常监测流程

1. 用户佩戴背带并打开 App。
2. App 自动连接外设并进入监测状态。
3. 用户姿态正常时，仪表盘保持绿色状态并记录良好时长。
4. 发现异常姿态持续超过阈值后，外设轻震提醒。
5. App 展示问题类型、持续时间、建议动作。
6. 用户坐直或完成微运动后，异常状态解除。
7. 当天结束后，App 生成日报。

### 6.3 每周复盘流程

1. App 提醒用户进行周评估。
2. 用户按上次角度拍摄侧面照。
3. AI 对比本周与上周关键指标。
4. App 展示改善项、未改善项和训练计划调整。
5. 用户可保存、分享或导出报告。

### F2. 首次校准与 AI 视觉体态评估

优先级：P0
目标：建立用户个人姿态基线，并用视觉 AI（云端 Qwen3.6-Plus）修正佩戴偏差。

#### 功能说明

- App 提供拍摄引导：侧身、自然站立、光线充足、身体关键点可见。
- AI 根据照片输出体态观察，不使用诊断性措辞。
- 评估项包括：头前倾、圆肩、胸椎后凸、骨盆前倾、左右倾斜。
- 评估结果转化为校准参数，用于修正 IMU 初始角度偏移。
- 用户可以跳过照片评估，使用纯 IMU 基线校准。
- **AI 模型**：云端 Qwen3.6-Plus API（初赛 P0）。网络不可用时降级到 3 套预置结果。
- **云端 API 端点、鉴权**：D1-D2 确定，需赛事提供的 API Key 或通义千问开放平台。

#### AI 输出规范

AI 文案必须使用辅助性表达：

- 可以说：「观察到头部相对肩线前移，建议关注颈部回缩训练」。
- 不可以说：「你患有上交叉综合征」或「需要治疗」。

#### 坐标系与节点参考系（落地实现细节）

为消除算法与固件之间的歧义，统一以下约定：

- **世界坐标系 W**：ENU（东-北-上），重力方向为 -Z。App 启动时由 BNO085 的「重力向量」自动估计。
- **节点局部坐标系 N_i**（i ∈ {C, T, L}）：BNO085 出厂约定的「设备坐标系」，X 右、Y 前、Z 上。
- **节点安装朝向**：背带佩戴后，节点局部 Z 轴指向天花板，X 轴指向用户左肩。固件启动时强制把 BNO085 设为「带重力校准」模式，开机 3 秒内完成水平自检。
- **姿态表达**：节点 i 输出的四元数 q_i 表示「从 N_i 到 W 的旋转」。

#### 关键角度定义


| 角度        | 定义                          | 物理意义        |
| --------- | --------------------------- | ----------- |
| NeckPitch | Node-C 相对 Node-T 在 W 系下的俯仰差 | 头前倾程度       |
| ThorPitch | Node-T 在 W 系下的俯仰            | 胸椎后凸        |
| LumPitch  | Node-L 在 W 系下的俯仰            | 腰椎塌陷 / 骨盆前倾 |
| TrunkRoll | Node-C 在 W 系下的横滚            | 左右倾斜        |
| TrunkYaw  | Node-C 与 Node-L 偏航差         | 脊柱侧弯趋势      |


所有角度单位：度。固件上行 BLE 仍发送原始四元数，App 端用上述定义计算角度。

#### 校准数学

**初赛版（5 秒快速校准）**：

设校准阶段采集 M=250 个样本（50Hz × 5 秒），全程采集「自然坐姿」。

- **零位偏置 Δq_i**：用全部样本四元数均值取共轭，作为节点 i 的零位偏置，后续所有四元数先乘以 Δq_i 再参与计算。
- **基线角度集合 B = {ThorPitch_0}**：取中位数作为用户基线（初赛仅 Node-T）。
- **基线稳定性指标 σ**：ThorPitch 标准差。若 σ_ThorPitch > 4°，提示「动得太多，请保持自然坐姿」并要求重采。
- **个人阈值** = 基线 + 偏置（ThorPitch +12°），初赛固定不可调。

**复赛版（30 秒完整校准）**：

设校准阶段采集 M=1500 个样本（50Hz × 30 秒），其中前 5 秒为「刻意稳定坐姿」校准段、后 25 秒为「自然坐姿」基线段。

- **零位偏置 Δq_i**：用校准段四元数均值取共轭，作为节点 i 的零位偏置。
- **基线角度集合 B = {NeckPitch_0, ThorPitch_0, LumPitch_0, TrunkRoll_0, TrunkYaw_0}**：取基线段中位数作为用户基线（3 节点）。
- **基线稳定性指标 σ**：基线段各角度标准差。若 σ_NeckPitch > 4° 或 σ_ThorPitch > 4°，提示重采。
- **个人阈值** = 基线 + 偏置，初始偏置按 §5.5.2 默认值；用户在设置页可手动微调 0-50%。

视觉评估结果作为「可选」校准修正量：在纯 IMU 基线之上，把头部前移程度作为一个一次性偏置加到 NeckPitch 基线上，避免不同佩戴习惯导致基线偏移。

| --- | --------------------------------------- | ------ |
| App | 拍摄引导、照片预览、重拍、确认上传/本地处理流程                | 移动端    |

### F3. 实时姿态监测与提醒

优先级：P0  
目标：识别持续不良姿态，并通过外设和 App 给出低干扰提醒。

#### 功能说明

- 从 3 节点 IMU 四元数计算关键角度。
- 识别姿态类型：正常、头前倾、驼背、腰部塌陷、左右倾斜、频繁扭转。
- 短暂动作不提醒，异常姿态持续超过阈值才提醒。
- 阈值基于用户基线动态调整，避免过度打扰。
- 外设提供震动提醒，App 展示具体原因和建议。

#### 姿态判定初始规则


| 姿态类型 | 初始判定逻辑                 | 默认触发时长    | MVP 级别 |
| ---- | ---------------------- | --------- | ------ |
| 头前倾  | NeckPitch 超过个人基线 + 10° | 5 分钟      | **P0** |
| 驼背   | ThorPitch 超过个人基线 + 12° | 5 分钟      | **P0** |
| 腰部塌陷 | LumPitch 低于个人基线 - 10°  | 5 分钟      | P1     |
| 左右倾斜 |                        | TrunkRoll | 超过 8°  |
| 久坐   | 连续佩戴超过 45 分钟           | 45 分钟     | P1     |


> MVP 触发提醒的判定只走头前倾 + 驼背两条路径，UI 提示文案统一为「你的坐姿开始驼了/头前移了」。其他类型在仪表盘上以「参考数据」形式可见，但不主动提醒。

#### 姿态状态机

```
        ┌──────────────┐
        │   NORMAL     │◀────────────────┐
        └──────┬───────┘                 │
   进入异常持续 ≥ 触发时长                 │ 恢复正常 ≥ 5s
               ▼                          │
        ┌──────────────┐  持续延长    ┌──────────────┐
        │  SOFT_ABN    │────────────▶│  HARD_ABN    │
        │ (注意 1-3 分钟)│            │ (持续异常)   │
        └──────────────┘             └──────┬───────┘
                                            │ 触发提醒
                                            ▼
                                     ┌──────────────┐
                                     │  ALERT_SENT  │── 10 分钟冷却 ──▶ ALERT_COOLDOWN
                                     └──────────────┘
```

**状态定义**：


| 状态             | 含义                 | 颜色     |
| -------------- | ------------------ | ------ |
| NORMAL         | 各角度在基线 ± 个人阈值内     | 绿      |
| SOFT_ABN       | 超过个人阈值，但持续未到触发时长   | 黄      |
| HARD_ABN       | 持续超过个人阈值 ≥ 触发时长    | 红      |
| ALERT_SENT     | 已发出提醒（震动 + 卡片）     | 红 + 脉冲 |
| ALERT_COOLDOWN | 冷却中，10 分钟内不重复提醒同类型 | 黄      |


**迁移条件**：

- NORMAL → SOFT_ABN：任一类型角度超过个人阈值。
- SOFT_ABN → HARD_ABN：连续在 SOFT_ABN 停留 ≥ 该类型触发时长。
- SOFT_ABN → NORMAL：连续 5 秒全部角度回到基线 ± 0.7 × 个人阈值。
- HARD_ABN → ALERT_SENT：触发外设震动 + App 提醒卡片。
- ALERT_SENT → ALERT_COOLDOWN：发送完成进入冷却。
- ALERT_COOLDOWN → NORMAL：角度恢复 ≥ 5 秒。
- 任何状态 → NORMAL：BLE 断连 ≥ 10 秒或校准失效。

**边缘条件**：

- 异常类型切换：若新类型严重程度 ≥ 当前类型，状态保持 HARD_ABN；否则降到 SOFT_ABN 重新计时。
- 频繁小幅抖动：滑动窗口内若 SOFT_ABN ↔ NORMAL 切换 ≥ 5 次 / 分钟，判定为「坐立不安」，不发提醒但记录到日报。
- 校准失效：若基线标准差 σ 在运行 5 分钟后明显恶化（> 8°），提示用户重新校准。
- 久坐与其他异常并行：久坐提醒的优先级最低，只有在 NORMAL 状态下才触发。

| 模块  | 任务                      | 负责人建议 |

### F8. 端侧 TFLite 姿态分类器（备选方案）

优先级：P1（MNN 集成遇阻时的回退方案）
**状态**：F9（端侧 Qwen MNN）为首选方案。若 MNN 集成在 D5 前遇阻（内存不足、延迟过高、框架兼容问题），回退到本方案。
目标：在手机端实时完成姿态状态分类，0 上传、0 延迟。

#### 功能说明

- **任务**：5 分类（正常 / 头前倾 / 驼背 / 腰椎塌陷 / 左右倾斜）。
- **输入**：5 角度滑窗（NeckPitch / ThorPitch / LumPitch / TrunkRoll / TrunkYaw），1 秒窗口，5 帧 @ 200ms。
- **输出**：5 类 softmax 概率分布。
- **模型大小**：≤ 300KB（INT8 量化后）。
- **部署位置**：Android 端 `assets/ml/posture.tflite`，TFLite Interpreter 加载。
- **推理节奏**：与 IMU 采样同步，每 200ms 一次。
- **用途**：作为 §F3 状态机的独立"判定源"，与规则化阈值互为冗余。

#### 研发任务

| --- | ------------------------------ | --- |
| 算法  | TFLite 模型训练与量化                 | 算法  |
| App | TFLite Interpreter 集成（Android） | 移动端 |


#### 验收标准

- 在预置的 4 类演示序列上，top-1 准确率 ≥ 90%。
- 单次推理时延 ≤ 20ms（Android 主流机型 ≥ 骁龙 7 系）。
- 推理崩溃时自动回退到规则化阈值，不影响主流程。

### F9. 端侧 Qwen 模型部署（MNN 推理）

优先级：P0（赛事硬要求：核心交互逻辑支持本地运行）
**状态**：初赛必做。赛事推荐技术栈 MNN + Qwen3.5 系列。
目标：在手机端通过 MNN 框架运行 Qwen3.5-2B 模型，实现姿态分类与本地反馈文案生成，核心推理 100% 离线可用。

#### 功能说明

- **模型**：Qwen3.5-2B（INT4 量化），初赛首选。复赛可升级 Qwen3-VL-2B（带视觉能力）。
- **框架**：MNN（深度适配 Arm SME2 指令集）。
- **部署位置**：Android 端 `assets/models/qwen3.5-2b-int4.mnn`。
- **推理管线**：
  1. 输入：BLE 传感器角度序列（NeckPitch / ThorPitch / LumPitch / TrunkRoll / TrunkYaw）+ 用户状态上下文
  2. MNN 推理：加载模型 → 输入编码 → 推理 → 输出解码
  3. 输出：姿态分类（5 类）+ 置信度 + 本地反馈文案
- **用途**：
  - 实时姿态分类（替代 F8 的 TFLite 分类器）
  - 本地反馈文案生成（如「头部有些前移，建议做颈部回缩」）
- **不做的事**：
  - 不做训练计划 AI 生成（初赛走预置，复赛再接端侧或云端，见 §5.8.2）
  - 不做视觉评估（走云端 Qwen3.6-Plus，见 F2）
  - 不做报告生成（走云端）
  - 不做端侧 Qwen3-VL-4B 以上模型

#### 量化与内存预算


| 模型               | 量化   | 模型大小    | 推理内存    | 目标延迟    | 最低设备            |
| ---------------- | ---- | ------- | ------- | ------- | --------------- |
| Qwen3.5-2B       | INT4 | ≈ 1.5GB | ≈ 2GB   | ≤ 1s    | 8GB RAM Android |
| Qwen3.5-0.8B（回退） | INT4 | ≈ 500MB | ≈ 800MB | ≤ 500ms | 6GB RAM Android |


#### 研发任务


| 模块    | 任务                            | 负责人   |
| ----- | ----------------------------- | ----- |
| AI 工程 | MNN 框架集成（Gradle 依赖 + .so 打包）  | 移动端   |
| AI 工程 | Qwen3.5-2B 模型下载与 INT4 量化      | AI 工程 |
| AI 工程 | 推理管线搭建（输入编码 → MNN 推理 → 输出解析）  | AI 工程 |
| App   | 姿态分类结果接入 §F3 状态机              | 移动端   |
| App   | 本地反馈文案渲染                      | 移动端   |
| QA    | 推理延迟测试（骁龙 7 系 / 8 系各 1 台）     | 测试    |
| QA    | 内存占用测试（8GB / 12GB 设备各 1 台）    | 测试    |
| 文档    | 模型版本 + 输入输出 schema + MNN 集成说明 | AI 工程 |


#### 验收标准

- MNN 框架成功加载 Qwen3.5-2B INT4 模型。
- **推理延迟**：首 token ≤ 1.5s，完整反馈文案（含 token 生成）≤ 3s（骁龙 7 系以上）。如 2B 模型不达标，回退到 Qwen3.5-0.8B INT4（目标 ≤ 1.5s 完整推理）；如 0.8B 仍不达标，回退到 F8 TFLite 规则引擎。
- 推理过程不产生网络请求（抓包验证）。
- 推理内存峰值 ≤ 3GB（8GB RAM 设备不 OOM）。
- 推理崩溃或超时（> 5s）时自动回退到规则化阈值，不影响主流程。
- 本地反馈文案无医疗化表述：模型输出结构化字段 → 本地查表翻译 → 禁词检查（详见[技术规格文档 §3.2](技术规格文档.md#32-安全限制安全链)端侧路径）。

### 11.3 设计原则

- 先让用户知道「现在是否需要调整」，再展示复杂数据。
- 医疗感要克制，避免制造焦虑。
- 状态颜色必须稳定一致：绿色正常、黄色注意、红色需要处理。
- 训练动作文案必须短、明确、可执行。
- 拍摄和佩戴引导要依赖图示，不依赖长文本。

**v1.4 新增硬规则**（违反即视为 P0 设计缺陷）：

- **震感克制**：外设单次震动 < 200ms；同类型异常 10 分钟内不重复；震动强度三档可调（弱 / 中 / 强），默认弱；连续震、节律震、长震均禁用。
- **状态色克制**：大面积红色只用于"持续异常"提醒和日报"未达标"提示；正常 / 轻微异常 / 单次提醒的视觉色用绿 / 黄 / 浅红，权重逐级降。
- **零位校准思维**：校准的本质是"用户当前的自然坐姿 = 基准"；不做复杂的"标准坐姿 vs 当前坐姿"对比，不让用户觉得自己"姿态不标准"。

以下问题不阻塞 MVP 启动，但会影响产品化方案：

1. Qwen3.5-2B MNN 推理在目标机型（骁龙 7 系以上）的实际延迟和内存占用（D5 前需 benchmark）。
2. 云端 Qwen3.6-Plus API 的鉴权方式和配额限制（D1-D2 确认，需赛事提供的 API Key 或通义千问开放平台）。
3. MNN 框架是否需要额外的 .so 库打包到 APK，以及 Arm SME2 指令集在目标设备上的支持情况。
4. App 首发平台是否只做 Android；若包含 iOS，BLE 和模型部署需要重新评估。
5. 外设形态是背带、贴片还是衣物集成；MVP 建议先用背带。
6. 是否需要康复师后台；MVP 不做。
7. 是否要申报医疗器械；当前 PRD 按健康辅助产品处理。
8. 云端 Qwen3.6-Plus API 月度调用预算上限？需确认赛事提供的免费额度或自费预算。
9. 端侧模型更新机制：OTA 推送 vs App 内下载？MVP 阶段固定打包，复赛需评估。
