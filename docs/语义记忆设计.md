# 语义记忆（教练"懂你"）设计方案 · 讨论稿

> 版本：2026-06-18（草案，待讨论后执行）
> 关联：[模型与记忆个性化设计](./模型与记忆个性化设计.md)（§3 把记忆分两层）
> 定位：本稿只设计**语义记忆层**（教练跨会话记住用户偏好/主诉/有效动作）。**习惯数值预测**是另一层（time×weekday 直方图），不在此稿。

---

## 1. 目标与边界
- **目标**：让端侧教练的文案随"对这个用户的了解"个性化——记住他的偏好（别说教/要鼓励）、自述上下文（早上脖子僵）、哪些动作有效、设定的目标。
- **不是**：不做姿态判断（规则）、不做数值时段预测（统计层）、不做医疗诊断。记忆只影响**语气与上下文**，最终文案仍过禁词/安全链。
- **隐私**：健康相邻 + 自述不适 = 敏感。**默认仅本地存储**（设备 FS），不上云；TTL 过期；用户可查看/清空。

## 2. 记什么（记忆 schema，借 catui-mem 的 7 类）
| 类型 | 姿态场景的内容 | 例 |
| --- | --- | --- |
| preferences | 教练风格、提醒容忍度 | "偏好鼓励式，别说教"；"午休时段别提醒" |
| struggles | 反复出问题的部位 | "颈前倾反复出现"；"右侧塌肩多" |
| lessons | 什么有效 | "起身活动后状态明显变好" |
| patterns | 语义化习惯（区别于数值层） | "开会时容易驼背" |
| decisions | 目标/承诺 | "目标：每天不驼背 > 4h" |
| knowledge | 环境事实 | "用升降桌"；"椅子偏低" |
| entities | 称呼 | "叫他小宇" |

> 取这 7 类是为了与 catui-mem 概念对齐，便于后续升级；**初赛只用 preferences / struggles / lessons 三类即可**。

## 3. 记忆从哪来（本项目的关键难点）
catui-mem 原生从**对话文本**抽取记忆，但本 App 不是聊天机器人、用户很少打字。所以来源要重新设计，**多数是结构化写入，不是从自由文本抽取**：
1. **Onboarding/资料**：首启 2-3 个轻问题（"想被怎样提醒？""最近哪里不舒服？"）→ 直接写 preferences/struggles。
2. **反馈信号**：建议卡 / 跟练完成上的 👍👎 / "有用吗" → 写 lessons（哪条/哪动作有效）。
3. **统计层 → 语义巩固**：把习惯直方图周期性总结成一条语义 pattern（"工作日午后易驼背"）。
4. **（可选）和教练说一句**：一个轻量自由输入框 → 这才需要抽取（heuristic 或端侧模型）。
> **设计取舍**：因为来源多为结构化，**catui-mem 的"抽取"价值不大，真正用到的是它的"评分/衰减/相关性检索/注入格式"**。这决定了下面的接入方式。

## 4. 怎么存（catui-mem 落地现实 → 两套方案）
**现状**：catui-mem `store.ts` 硬编码 `node:fs/node:path`，且"所有模块都走它"；ESM + node 内建。RN/Hermes 没有 node 内建。

- **方案 A（推荐，初赛）：RN 原生轻量语义记忆**
  借 catui-mem 的**设计**（7 类、recency/importance/relevance 评分、遗忘衰减、注入格式），但用我们自己**精简实现** + `expo-file-system` 持久化。可控、零 node 依赖、体积小。鉴于来源多为结构化，不需要它的重抽取/巩固管线。
- **方案 B（复赛）：完整移植 catui-mem**
  Metro 别名把 `node:fs`→`expo-file-system` 适配器、`node:path`→`path-browserify`，直接吃整包。拿到 linking/consolidation/insights/间隔重复。但集成风险高（要核对它没有其它 node-only 依赖如 crypto）。

> 建议：**A 先落地拿到效果**，把数据结构与 catui-mem 对齐；**B 留复赛**升级，迁移成本低。

## 5. 怎么用（注入教练 prompt）
- 写入：上面 4 个来源 → 记忆条目。
- 读取：`adviceOrchestrator.buildPrompt` 前，按**当前姿态 tags** 检索 top-K 最相关记忆 → 拼一小段"关于这个用户"前缀。
- **硬约束**：端侧 0.5B 上下文小、且要短 prompt 保 TPS → 注入**最多 1-3 条、每条 ≤20 字**，按 relevance×recency×importance 排序取顶。
- 例：记忆"偏好鼓励式" + "颈前倾反复" → 注入 `已知:他偏鼓励、颈前倾反复` → 文案更对味。

## 6. 边/云与抽取
- **结构化写入**（来源 1/2/3）：无需 LLM，纯本地、离线。
- **自由文本抽取**（来源 4，可选）：heuristic 规则即可；要更准用端侧 Qwen 或云端低频抽取。
- **巩固/洞察**：复赛可用云端 LLM 周期性把碎片记忆合并成画像。

## 7. 隐私与用户控制
- 仅本地 `documentDirectory/memory/`；不默认上云。
- 每条带 TTL/衰减；低分自动淘汰。
- **记忆管理视图**（Settings 内或"我的"）：列出"教练记住的事"，可单条删除 / 一键清空。PII 过滤（去手机号等）。
- 记忆**不得**变成医疗结论；注入内容仍过禁词链。

## 8. 架构与接入点
```
src/posture/memory/
  types.ts            # MemoryItem / MemoryType / 评分字段
  store.ts            # expo-file-system 持久化（load/save，节流）
  scoring.ts          # recency×importance×relevance + 衰减（借 catui-mem 公式）
  service.ts          # remember()/retrieve(tags,k)/inject()/forget()/clearAll()
写入钩子：Onboarding、反馈 👍👎、TrainingScreen 完成、growth 周期巩固
读取钩子：adviceOrchestrator.buildPrompt 注入 service.inject(tags)
UI：Settings「教练记忆」管理卡
```

## 9. 数据模型（草案）
```ts
type MemoryType = 'preference' | 'struggle' | 'lesson' | 'pattern' | 'decision' | 'knowledge' | 'entity';
interface MemoryItem {
  id: string;
  type: MemoryType;
  text: string;            // ≤40 字，已脱敏
  tags: string[];          // 如 ['TECH_NECK','tone']，供相关性检索
  importance: number;      // 0-1
  createdAt: number;
  lastUsedAt: number;
  uses: number;
  source: 'onboarding'|'feedback'|'consolidation'|'chat';
  ttlDays?: number;
}
```

## 10. 阶段化
| 项 | 初赛（lite，方案 A） | 复赛 |
| --- | --- | --- |
| 类型 | preference/struggle/lesson | 全 7 类 |
| 来源 | onboarding + 反馈 👍👎 | + 统计巩固 + 自由文本抽取 |
| 引擎 | RN 原生精简（FS 持久化） | 完整 catui-mem 移植（Metro 别名 + FS 适配器） |
| 注入 | top-1~3 极简前缀 | 加权检索 + 链接 |
| 抽取 | 无（结构化直写） | heuristic / 端侧 / 云端 |
| 巩固/洞察 | 无 | 云端周期巩固成画像 |

## 11. 待和你讨论的决策
1. **引擎选型**：A（RN 原生轻量，初赛落地）还是直接冲 B（完整 catui-mem 移植）？
2. **初赛范围**：要不要真接进 demo（需要加 onboarding + 反馈 👍👎 的小 UI），还是先把"记忆→注入"链路打通、UI 留后？
3. **记忆来源优先级**：先做哪几个来源（建议 onboarding + 反馈）。
4. **注入预算**：0.5B 上注入几条/多少字（建议最多 3 条、总 ≤60 字）。
5. **隐私基线**：确认"仅本地、不上云、可清空"，以及是否需要首启告知。
